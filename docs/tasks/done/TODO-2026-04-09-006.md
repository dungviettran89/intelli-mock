**Completed:** 2026-04-09

## Context
Phase 2 (Mock CRUD + Matching) is the next phase after Phase 1 (Foundation) completion. The route matcher is the first component to build — it's the foundation for mock endpoint routing, sample management, and the entire mock/auto endpoint pipeline. Without it, incoming requests to `/_it/mock/*` and `/_it/auto/*` cannot be matched to the correct MockEndpoint.

The matcher must implement longest-match algorithm with wildcard support as specified in ARCHITECT.md, scoped by tenant.

## Steps
1. **Create RouteMatcher** — implement `match(tenantId, method, path)` that:
   - Takes an array of MockEndpoint, filters by tenantId + method
   - Scores each endpoint's `path_pattern` against the request path
   - Supports wildcards: `*` (match segment), `**` (match rest), `:param` (capture)
   - Returns the longest-match winner + extracted params, or null
2. **Create MockService** — implement CRUD operations:
   - `create(tenantId, data)` — create MockEndpoint
   - `findAll(tenantId)` — list all endpoints for tenant
   - `findById(tenantId, id)` — get single endpoint
   - `update(tenantId, id, data)` — update endpoint
   - `delete(tenantId, id)` — soft delete endpoint
   - Uses TypeORM Repository with `tenantId` on every query
3. **Write unit tests for RouteMatcher** — test cases:
   - Exact match (`/api/users` matches `/api/users`)
   - Longest match (`/api/users/:id` wins over `/api/*` for `/api/users/42`)
   - Wildcard `**` matches rest of path
   - Wildcard `*` matches single segment
   - Param extraction (`:id` → `{ id: "42" }`)
   - No match returns null
   - Tenant isolation (same path, different tenant → no match)
4. **Write unit tests for MockService** — test cases:
   - CRUD operations with mocked TypeORM repository
   - All queries include `tenantId` filter
   - Delete sets status to 'deleted' (not hard delete)
   - Uses DI container with `@injectable()` decorator
